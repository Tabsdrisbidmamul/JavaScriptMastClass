                                                                    Brian Mearns 2 years ago

                                                                    Even though this is commonly how it is described, I think it's misleading to say that await _pauses_ execution; this implies that your function is executing synchronously with a big blocking `pause()` in the middle like you might see in a multithreaded language like Java.

                                                                    In actuality, The `await` keyword breaks the synchronous flow and creates a new promise handler from every subsequent statement in the function; this handler executes asynchronously. It is triggered by the settling of the promise, but this is happening asynchronously, not in a blocking manner.

                                                                    The distinction is subtle for anyone not too familiar with the JavaScript execution model, but it's an important distinction for breaking JS programmers out of a multithreaded frame of mind, which can lead to confusing behavior and poor performance.

                                                                        Charbugs Brian Mearns 2 years ago

                                                                        I agree with you and I want to add that this isn't just a "subtle distinction" but an important one:

                                                                         async function fn() {
                                                                          let result = await Promise.resolve("foo") 
                                                                          console.log(result)
                                                                        }

                                                                        fn()
                                                                        console.log("bar")

                                                                        Here you will see "bar" before "foo" on the console. I.e. await will stop the execution of the function and continues when the promise is settled. But this won't happen until the entire current call stack is processed. This concept is known as "run to completion".

                                                                                                J.P. Duvet 6 months ago

                                                                                                As many other articles on this topic, the description of "await" is very misleading.

                                                                                                There is a contradiction between the 2 following statements:
                                                                                                1. The word async before a function means one simple thing: a function always returns a promise
                                                                                                2. The keyword "await" makes JavaScript wait until that promise settles and returns its result

                                                                                                Why an async function would always return a promise if "await" pauses the execution waiting for a result?
                                                                                                It should then logically return the result and not a promise.

                                                                                                An async function doesn't return directly a result because, indeed, there no "execution pause" but an implicit promise creation with the statements immediately following "await". The async function immediately returns this promise, the processing continues and this implicit promise will settle later.

                                                                                                To see it properly (as a "callback function" for ex.) can save a lot of time in debugging dream-like "synchronous" code...

                                                                                                    One note on the last example: the task says "wait to get 10" but that's actually impossible.                                                                                                                       Kris Dover jugal joshi a year ago

                                                                                                                      I had a similar question regarding why would anyone add async/await to an existing function which only returns a single Promise which doesn't use then() chaining or catch() error handling e.g.


                                                                                                                      // why do this?
                                                                                                                      async function f() {
                                                                                                                      return await new Promise((resolve, reject) => { ... });
                                                                                                                      }

                                                                                                                      // as opposed to this
                                                                                                                      function f() {
                                                                                                                      return new Promise((resolve, reject) => { ... });
                                                                                                                      }

                                                                                                                      I think the answer is there is no benefit since:

                                                                                                                      1. your return value is already a Promise so there is nothing for async to wrap,
                                                                                                                      2. there is no intermediate result to wait on or promise de-chaining/semantic improvement to be had and
                                                                                                                      3. we don't need to handle errors from the rejected promise in the function.

                                                                                                                      So IMO you're absolutely correct. The reason the Promise.all() example uses async/await is clearly mentioned in the article, and has to do with the 3rd point I just mentioned:


                                                                                                                      In case of an error, it propagates as usual: from the failed promise to Promise.all, and then becomes an exception that we can catch using try..catch around the call.
